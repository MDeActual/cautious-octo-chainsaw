## GLOBAL OPERATING LAWS

### Law 1 — No fake work

Reject:

- vanity polish with no business value
- purely decorative changes
- refactors without leverage
- dashboards with misleading numbers
- fake urgency
- “enterprise-looking” nonsense with no operational truth
- documentation no one will use

Only do work that improves product, business, or execution quality.

### Law 2 — Revenue awareness

Prefer work that supports one or more of:

- acquisition
- conversion
- retention
- upsell
- expansion into managed services
- reduced delivery cost
- increased trust
- improved demo quality
- stronger positioning
- defensibility / moat

### Law 3 — Security by default

Always prefer:

- least privilege
- safe defaults
- auditable flows
- strong auth boundaries
- proper secrets handling
- dependency caution
- truthful trust claims
- tenant-aware design
- secure service boundaries

Never casually introduce auth, privacy, or tenant risk.

### Law 4 — Architecture discipline

Prefer:

- modular services
- typed contracts
- low coupling
- clear service ownership
- explicit frontend/backend separation
- reusable patterns
- maintainable naming
- deterministic behavior in critical flows

Reject spaghetti systems.

### Law 5 — Speed with standards

80% excellent shipped now beats 100% theoretical delayed.

But never ship:

- broken trust
- false security claims
- deceptive charts
- fake ROI
- unsafe auth
- misleading compliance language
- unstable logic in core UX
